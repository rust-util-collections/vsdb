## Findings

### [HIGH] engine & raw KV: safe deserialization/recovery bypasses prefix reconstruction contract

**WHERE**: `core/src/common/engine/mod.rs:421-436`; `core/src/basic/mapx_raw/mod.rs:81-87,384-397,461-463`
**WHAT**: `MapxRaw::from_bytes` documents raw prefix reconstruction as unsafe and valid only for bytes produced by `as_bytes` from a same-version instance, but safe serde deserialization accepts any 8-byte prefix and reconstructs a handle. `from_meta` also reads raw bytes from disk and calls the unsafe reconstruction path without checking the length first, so truncated/corrupt metadata can panic in prefix copying instead of returning an error.
**WHY**: Violates technical-patterns.md 5.2, "from_bytes Deserialization of Untrusted Data". Safe APIs can construct a handle for an arbitrary prefix, after which normal `insert`/`remove`/`clear` mutate that namespace.
**FIX**: Do not expose raw-prefix reconstruction through safe `Deserialize` unless the bytes are proven to be VSDB-owned metadata. Validate exact metadata length and return `Err` in `from_meta`; consider feature-gating/removing safe handle deserialization or adding authenticated/registered metadata.

### [HIGH] typed-collections: `Mapx` accepts nondeterministic key encodings

**WHERE**: `strata/src/basic/mapx/mod.rs:63-83`; `strata/src/common/ende.rs:45-56,149-180`
**WHAT**: `Mapx<K, V>` only requires `K: KeyEnDe`, while `KeyEnDe` is blanket-implemented for serde types including `HashMap` and `HashSet`. Logically equal keys containing these types can serialize to different byte sequences, causing lookups with an equivalent key to miss.
**WHY**: Matches technical-patterns.md 6.1, "Postcard Encoding Non-Determinism": storage keys must have deterministic bytes because the encoded bytes are the database key.
**FIX**: Introduce and require a deterministic-key marker trait for `Mapx` keys, or otherwise prevent nondeterministic serde containers from satisfying public key bounds. At minimum, document this hard precondition directly on `Mapx`.

### [HIGH] versioning: recovered dirty state can be cleared before crash repair

**WHERE**: `strata/src/versioned/map.rs:191-205,274-276,1087-1095,1168-1195`
**WHAT**: Deserialization/from_meta rebuilds B+ tree node ref-counts but not commit ref-counts. Commit repair only happens later in `gc()` if `gc_dirty` is true. A normal operation that calls `decrement_ref()` sets `gc_dirty = true` for its local cascade and then unconditionally clears it, so it can erase a dirty marker recovered from a prior interrupted commit/rollback/merge/delete before global repair runs.
**WHY**: Violates technical-patterns.md 2.1 reference-count balance and 2.5 dirty-flag crash recovery. Clearing the recovered dirty marker can leave stale commit ref-counts or orphaned commits unrepaired.
**FIX**: On deserialization/from_meta, run commit-level repair before returning when `gc_dirty` is true, or make every mutating public operation repair a pre-existing dirty state before local ref-count changes. `decrement_ref()` should preserve an already-dirty state instead of unconditionally clearing it.

### [HIGH] Merkle tries: SMT proof verification is not bound to the expected key

**WHERE**: `strata/src/trie/smt/proof.rs:156-187`; `strata/src/trie/mod.rs:384-386`; `strata/src/trie/proof.rs:302-305`
**WHAT**: SMT proof verification accepts only `(root_hash, proof)` and derives the path from `proof.key_hash`. It never accepts or checks the key the caller intended to verify, so a valid proof for key A verifies against the root even when the caller expected key B unless every caller independently compares the key hash.
**WHY**: Violates technical-patterns.md 3.1 proof generation/verification soundness: a proof for a different key must fail. The MPT verifier has an expected-key binding, but SMT does not.
**FIX**: Change SMT verification APIs to accept the expected raw key or expected key hash, compare it with `proof.key_hash` before walking siblings, update `VerMapWithProof<_, _, SmtCalc>::verify_proof`, and add a wrong-key regression test.

### [HIGH] slot index: mutations after `save_meta()` can run with a clean dirty flag

**WHERE**: `strata/src/slotdex/mod.rs:225-288,299-363`
**WHAT**: `save_meta()` clears the dirty bit, but later `insert`, `remove`, and `clear` use `dc::inc`, `dc::dec`, and `dc::zero`, which preserve the current dirty state instead of marking the counter dirty before persistent mutations. A crash between data/tier writes and counter maintenance can leave a clean-but-stale counter, causing recovery to skip rebuilding.
**WHY**: Matches technical-patterns.md 2.5, "Dirty Flag Not Set on Crash": dirty metadata must be set before persistent mutation begins and cleared only after a clean checkpoint.
**FIX**: At the start of every mutating public entrypoint, set the dirty bit before any persistent map/orphan mutation and leave it dirty until the next successful `save_meta()`.

### [HIGH] DAG collections: ID ceiling rename is not crash-durable

**WHERE**: `strata/src/dagmap/mod.rs:75-85`; `strata/src/dagmap/raw/mod.rs:126-136`
**WHAT**: `persist_ceiling()` writes and fsyncs a temp file, then renames it over `dag_id_ceiling`, but never fsyncs the parent directory after the rename. A crash after `rename` returns can lose the directory entry update and recover the old ceiling, allowing already-returned IDs to be reissued.
**WHY**: Violates dagmap.md INV-DG1, ID counter monotonicity after crash, and the technical-patterns.md dirty/crash-safety class. Duplicate IDs can overwrite parent `children` entries; the duplicate guard is debug-only.
**FIX**: After `fs::rename(&tmp, &self.path)`, open the parent directory and call `sync_all()` before returning any ID from the new batch.

### [HIGH] DAG collections: empty encoded values become deletion tombstones

**WHERE**: `strata/src/dagmap/rawkey/mod.rs:179-184`; `strata/src/dagmap/raw/mod.rs:205-213,247-270`
**WHAT**: `DagMapRawKey<V>::insert` stores `value.encode()` into `DagMapRaw`. The raw layer treats an empty stored value as `None`, and `remove` writes an empty value as the tombstone. Any valid value type whose encoding is empty cannot round-trip through the typed API, and the raw insertion guard is debug-only.
**WHY**: Violates technical-patterns.md 6.3/6.1 encoding round-trip expectations and review-core's `decode(encode(x)) == x` invariant for values stored through public APIs.
**FIX**: Stop using raw empty bytes as an in-band deletion marker, or make insertion fallible and reject empty encodings for both raw and typed DAG APIs in all builds.

### [HIGH] vector index: pruning can leave stale one-way reverse edges

**WHERE**: `strata/src/vecdex/mod.rs:450-469,670-693`; `strata/src/vecdex/hnsw.rs:258-277`
**WHAT**: `prune_neighbors()` rewrites a node's complete neighbor list after selecting at most `m_max` entries. Current callers repair bidirectionality only for the newly-added edge being considered. If pruning evicts a pre-existing third-party neighbor, that evicted neighbor can still retain the reverse edge, leaving a one-way adjacency.
**WHY**: Violates vecdex.md INV-VD2, bidirectional edge consistency. This is a concrete multi-key logical update inconsistency: deleting an edge requires updating both adjacency lists.
**FIX**: Make `prune_neighbors()` return evicted neighbor IDs, or have each caller capture before/after around every prune and remove the pruned node from all evicted neighbors' lists.

### [MEDIUM] B+ tree: `bulk_load` can create duplicate keys in an ordered map

**WHERE**: `strata/src/basic/persistent_btree/mod.rs:930-948`
**WHAT**: `PersistentBTree::bulk_load` accepts pre-sorted `(key, value)` entries and packs them directly into leaf nodes without validating strict ordering or coalescing duplicates. Sorted duplicate keys can produce leaves with duplicate map keys.
**WHY**: Violates technical-patterns.md 1.2 and btree.md INV-BT2, which require strict key ordering. The observable behavior can diverge from map semantics: iteration can return the same key twice, and removal may remove only one duplicate.
**FIX**: Validate strictly increasing keys before allocation, or normalize input by keeping one value per key before packing leaves.

### [MEDIUM] typed-collections: bounded range scans can walk the entire prefix tail

**WHERE**: `strata/src/basic/mapx_ord_rawkey/mod.rs:106-119,173-192`; `core/src/common/engine/mmdb.rs:250-278`; `core/src/basic/mapx_raw/mod.rs:190-191`
**WHAT**: `MapxOrdRawKey::range` delegates to `MapxRaw::range`, whose engine computes the requested upper bound but uses the prefix successor as the underlying iterator end hint, enforcing the requested high bound only with a lazy filter. Consuming a small bounded range can therefore scan from the lower bound through the end of the prefix; `get_le(..)` uses `range(..=key).next_back()` and can scan backward from the prefix end.
**WHY**: Violates review-core's hot-path performance checks for iterator/range operations. This is a concrete O(prefix-tail) amplification for public range/predecessor APIs.
**FIX**: Pass the requested upper bound as the engine iterator end where possible, and for reverse/predecessor scans seek from the requested high bound instead of the prefix successor. Handle inclusive upper bounds with a byte successor or equivalent inclusive seek semantics.

### [MEDIUM] slot index: `clear()` leaves large per-slot containers' backing maps orphaned

**WHERE**: `strata/src/slotdex/mod.rs:361-372,743-749,786-793`
**WHAT**: `DataCtner::Large` stores entries in a separate `MapxOrd<K, ()>` and serializes only that handle plus length. `SlotDex::clear()` calls `self.data.clear()` directly, deleting the top-level serialized handles but never clearing the large containers' backing map prefixes.
**WHY**: Resource lifecycle leak analogous to technical-patterns.md 1.3 structural sharing leak: dead persisted data becomes unreachable and can accumulate unbounded disk usage.
**FIX**: Before clearing `self.data`, iterate existing `DataCtner` values and call `map.clear()` for every `DataCtner::Large`, then clear the top-level data and tiers.

### [MEDIUM] DAG collections: valid deep DAG chains are treated as corrupt

**WHERE**: `strata/src/dagmap/raw/mod.rs:205-225,281-297`
**WHAT**: `get()` stops after 1024 parent links and returns `None`, while `prune_mainline()` errors once the mainline exceeds 1024. Safe public APIs do not enforce a maximum depth at creation time, so a valid acyclic chain longer than 1024 becomes unreadable/unprunable.
**WHY**: Violates dagmap.md INV-DG2 reachability for acyclic parent references. The cycle guard prevents infinite traversal, but it conflates valid deep chains with cycles.
**FIX**: Detect actual cycles using visited instance IDs, or enforce and document the maximum depth when creating children so unreachable ancestors cannot be created through safe APIs.

### [MEDIUM] DAG collections: public `destroy()` leaves parent child references dangling

**WHERE**: `strata/src/dagmap/raw/mod.rs:390-407`; `strata/src/dagmap/rawkey/mod.rs:216-220`
**WHAT**: `DagMapRaw::destroy` clears a node and its descendants but explicitly does not remove the node from its parent's `children` map. The doc says callers must remove `parent.children` first, but `children` is private and `DagMapRawKey::destroy` exposes the behavior without warning, so callers can destroy an attached child and leave its parent with a stale child entry.
**WHY**: Violates dagmap.md INV-DG3 orphan prevention. Internal pruning removes parent links before destroying children, but the public destroy API does not own the same invariant.
**FIX**: Have `destroy()` unlink itself from its parent, expose a parent-side removal API that owns the invariant, or restrict/document `destroy()` as valid only for detached/root nodes and mirror that contract in the typed wrapper.

### [LOW] encoding: dirty count helpers let count bit 63 corrupt the dirty flag

**WHERE**: `strata/src/common/dirty_count.rs:36-58`
**WHAT**: `set_count(raw, new_count)` computes `new_count | (raw & DIRTY_BIT)`, so `new_count >= 2^63` forces the dirty bit on regardless of `raw`. `inc()` has the same latent issue when the count crosses the reserved bit.
**WHY**: Violates the dirty-count split documented in the file: bit 63 is reserved for the dirty flag and bits 0..62 are the count.
**FIX**: Mask count inputs/results with `!DIRTY_BIT` and make `inc()` saturate or return an error before crossing `DIRTY_BIT - 1`.

### [LOW] vector index: public docs describe filtered-search beam expansion incorrectly

**WHERE**: `strata/docs/vecdex.md:103-104`; `strata/src/vecdex/mod.rs:598-602`
**WHAT**: The docs say filtered search expands `ef` to `ef * 2`, but the implementation uses `(ef * 4).max(k * 2)`.
**WHY**: Violates the doc-code alignment rule from review-core.md Phase 4.4 for public API behavior. Users tuning filtered search will rely on stale documentation.
**FIX**: Update the docs to match the implementation, or change the implementation if `ef * 2` is the intended contract.
